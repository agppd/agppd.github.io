## 一、二进制编码格式

依据 IEEE 754 标准，当浮点数的**指数位全部为 1** 且 **尾数位不全为 0** 时，该数即为 NaN。

以 32 位单精度浮点数（binary32）为例，其位模式如下：

$$
\underbrace{S}_{\text{符号位}}\;\underbrace{11111111}_{\text{指数位}}\;\underbrace{A\;xxx\ldots xxx}_{\text{尾数位（23 位）}}
$$

- **$S$（符号位）**：在比较操作中，正 NaN 与负 NaN 视为相等；符号位通常不影响 NaN 的核心语义。
- **指数位**：全部为 1（即 8 位全为 1）。
- **尾数位**：非零。

尾数部分的最高有效位 $A$ 决定了 NaN 的具体类型：

| $A$ 的值 | 类型                             | 行为特征                                |
|:------:| ------------------------------ | ----------------------------------- |
| 1      | **Quiet NaN（QNaN，安静 NaN）**     | 参与运算时不触发浮点异常，结果通常传播 QNaN。           |
| 0      | **Signaling NaN（SNaN，信号 NaN）** | 参与运算时触发“无效操作”（Invalid Operation）异常。 |

尾数中除 $A$ 以外的剩余位被称为 **payload**，可用于承载额外的诊断或状态数据。

> **注**：上述约定（$A=1$ 为 QNaN，$A=0$ 为 SNaN）被 x86、ARM、PowerPC 等主流架构采用。少数架构（如 PA-RISC）则采用相反的定义，在实际跨平台开发中需注意这一差异。

### 常见位模式示例

| 类型            | 单精度（binary32）十六进制范围         |
| ------------- | --------------------------- |
| 正无穷           | `0x7F800000`                |
| 负无穷           | `0xFF800000`                |
| Signaling NaN | `0x7F800001` ~ `0x7FBFFFFF` |
| Quiet NaN     | `0x7FC00000` ~ `0x7FFFFFFF` |

双精度（binary64）的对应范围为：

| 类型            | 双精度十六进制范围                                   |
| ------------- | ------------------------------------------- |
| Signaling NaN | `0x7FF0000000000001` ~ `0x7FF7FFFFFFFFFFFF` |
| Quiet NaN     | `0x7FF8000000000000` ~ `0x7FFFFFFFFFFFFFFF` |

## 二、QNaN 与 SNaN 的行为差异

### Quiet NaN（QNaN）

QNaN 被设计为“安静地”在计算中传播。当一个操作数包含 QNaN 时，运算结果通常为 QNaN，且**不会引发浮点异常信号**（除非异常屏蔽位被显式配置）。标准算术运算中，$ \text{QNaN} + 5.0 $ 的结果为 QNaN。

QNaN 通常由无效的数学运算自动产生，例如 $ 0.0 / 0.0 $、$ \sqrt{-1.0} $ 或 $ \log(-5.0) $。

### Signaling NaN（SNaN）

SNaN 被设计为在参与任何算术运算或浮点比较时，主动触发“无效操作异常”。这一特性使其在理论上适用于标记未初始化变量或缺失数据。

应用场景示例：

```c
double buffer[100];
// 将缓冲区初始化为 SNaN
for (int i = 0; i < 100; i++) {
    buffer[i] = make_snan();
}
// 若程序意外读取未初始化的元素并参与计算，将立即触发异常
```

然而，在实际应用中，SNaN 的使用率远低于 QNaN。主要原因包括：

1. **硬件支持差异**：部分通用处理器并未完整实现 SNaN 的异常触发机制，或默认将 SNaN 静默转换为 QNaN 处理。
2. **语言标准滞后**：C 和 C++ 等主流语言的标准直至 C99/C++11 后才逐步完善对 SNaN 操作的规范化支持。
3. **工具链限制**：编译器在处理 SNaN 时的行为尚未完全统一。

## 三、产生 NaN 的典型操作

IEEE 754 标准明确规定了若干会返回 NaN 的计算情形：

### 1. 涉及 NaN 的操作数

- 操作数包含 QNaN 时，结果通常传播该 QNaN。
- 操作数包含 SNaN 时，标准要求触发无效操作异常，并可选择返回一个 QNaN。

### 2. 不定式（Indeterminate Forms）

| 运算类型 | 示例                        |
| ---- | ------------------------- |
| 除法   | $ 0/0 $、$ \infty/\infty $ |
| 乘法   | $ 0 \times \infty $       |
| 加法   | $ \infty + (-\infty) $    |
| 减法   | $ \infty - \infty $       |

### 3. 定义域错误的实数运算

- 对负数开偶次方根：$ \sqrt{-1} $
- 对负数取自然对数：$ \log(-5) $
- 超出反三角函数定义域的输入

## 四、NaN 的运算特性与判定方法

本节系统阐述 NaN 在算术运算和比较中的行为规律，以及可靠判定 NaN 的常用技术。

### 算术运算特性

- **QNaN 的传播**：任何涉及 QNaN 的算术运算（加、减、乘、除、平方根、取余等）的结果通常为 QNaN。例如：
  $
  \text{QNaN} + 2.0 = \text{QNaN}, \quad \text{QNaN} \times 0 = \text{QNaN}
  $
  这里“0”与 QNaN 的乘法仍然返回 QNaN，即使数学上 \(0 \times \text{未定义}\) 仍是未定义。
- **SNaN 的触发**：涉及 SNaN 的运算会首先引发无效操作异常，若异常被屏蔽，则结果通常为 QNaN，且该 QNaN 的 payload 可能与原始 SNaN 不同（由具体实现决定）。

### 比较特性（IEEE 754 核心规定）

**NaN 与任何浮点数值（包括其自身）的比较结果均为“假”（false）**。具体表现为：

$
\text{NaN} = \text{NaN} \quad \text{为} \quad \text{false}
$

$
\text{NaN} \neq \text{NaN} \quad \text{为} \quad \text{true}
$

$
\text{NaN} < 0,\; \text{NaN} > 0,\; \text{NaN} \le 0,\; \text{NaN} \ge 0 \quad \text{均为} \quad \text{false}
$

此外，在无序比较（unordered comparison）中，如 `isunordered(x, y)` 会返回真，当且仅当任一操作数为 NaN。这一特性使得 NaN 不适合作为排序键或哈希键，且需要在循环条件中特别处理。

### 判定方法

判定一个浮点数是否为 NaN，常见方法包括：

1. **标准库函数 / 宏**（推荐）：
- C99 及之后提供 `isnan(x)` 宏，位于 `<math.h>`，通常映射到编译器内建函数（如 GCC 的 `__builtin_isnan`）。

- C++11 提供 `std::isnan(x)`，位于 `<cmath>`。

- 这些方法具有最佳的可移植性，且符合标准规定——它们作为非计算性操作，不应因 SNaN 而触发异常（在正确实现的工具链上）。
2. **利用自比较**（不推荐）：

3. $
   x \neq x \quad \text{当且仅当 } x \text{ 是 NaN}
   $

4. 因为只有 NaN 满足此性质。但该方法依赖于浮点比较，对于 SNaN 可能引发异常，且可读性差，故实际工程中应避免。

5. **位模式检查**（底层实现）：
   通过读取浮点数的二进制表示（如使用 `memcpy` 或 `union`），检查指数位是否全为 1 且尾数位非零。例如，对于单精度：
   
   ```c
   uint32_t bits = *(uint32_t*)&f; // 注意严格别名问题，实际应使用 memcpy
   if ((bits & 0x7F800000) == 0x7F800000 && (bits & 0x007FFFFF) != 0)
    // 是 NaN
   ```

或者，阅读如下来自 GCC 中 `__builtin_isnan` 的实现（启用 `-fsignaling-nans` 后）

```asm
fmov    w1, s0        ; 将浮点数 s0 的位模式传送到整数寄存器 w1
mov     w0, -16777216 ; 加载用于检查 NaN 的掩码
; -16777216 = 1111 1111, 1111 1111, 1111 1111, 1111 1111, 1111 1111, 0000 0000, 0000 0000, 0000 0000
; 联系 IEEE 754 不难理解为什么了
cmp     w0, w1, lsl 1 ; 将位模式左移一位后与掩码比较
cset    w0, cc        ; 根据比较结果设置返回值
```

这种方法完全避免了浮点异常，但可移植性依赖于字节序和浮点格式（IEEE 754），且需注意处理 SNaN 和 QNaN 的区分。

不同判定方式的优缺点如下：

| 方式           | 优点         | 缺点                                         |
| ------------ | ---------- | ------------------------------------------ |
| `isnan` 标准宏  | 可移植、符合标准   | 需要 C99/C++11 支持，部分编译器在 `-ffast-math` 下行为异常 |
| 自比较 `x != x` | 无需库支持      | 可能触发 SNaN 异常，可读性差                          |
| 位模式检查        | 无异常，速度可能更快 | 依赖实现细节，非标准，需处理字节序                          |

## 五、无效操作异常（Invalid Operation Exception）

IEEE 754 定义了五种基本浮点异常，其中“无效操作”与 NaN 直接相关。

当无效操作发生时，系统依据异常屏蔽位的状态执行以下动作：

1. **若异常未被屏蔽**：触发陷阱（trap），执行用户定义的信号处理程序。
2. **若异常被屏蔽（默认状态）**：设置对应的异常状态标志，并返回一个 **QNaN** 作为运算结果。

特别地，IEEE 754-2008 和后续修订版指出：**判断一个数是否为 NaN 的操作（如 `isNaN`）不应引发无效操作异常**。这类检测属于非计算性（non-computational）操作，不应受 SNaN 的影响。

在实际编译器中，该标准的实现存在演进过程。早期及部分架构的编译器可能使用浮点比较指令（如 \( x \neq x \)）实现 `isNaN`，由于浮点比较本身会触发 SNaN 异常，这构成了与标准字面要求的偏差。为此，部分编译器（如 GCC 在 AArch64 目标上）已调整为使用整数位运算来检测 NaN 的模式，从而严格避免浮点异常。但在 x86 等部分架构上，这一修复尚未完全覆盖，开发者需结合具体编译选项（如 `-fsignaling-nans`）进行验证。

## 六、编程中的注意事项

### NaN 的比较语义

如第四节所述，NaN 与任何值的比较均为假。因此在编写条件语句时，应避免直接依赖 `x == x` 或 `x != x`，而应使用 `isnan` 宏进行显式判断。

### 编译选项的影响

部分编译器提供忽略 NaN 语义的优化选项（如 GCC 的 `-ffast-math`，其中隐含 `-ffinite-math-only`）。启用该选项后，编译器假设程序不涉及 NaN 和无穷大，在此条件下 `isnan` 可能被常量折叠为固定值（如 0）。若代码中确实存在 NaN 处理逻辑，启用此类全局优化选项可能导致不正确的结果。

### Signaling NaN 的启用

如需利用 SNaN 触发异常的功能，GCC 提供了实验性的编译选项：

```bash
gcc -fsignaling-nans source.c
```

该选项告知编译器保留 SNaN 的操作语义，避免将其静默转换为 QNaN，并确保内建函数（如 `__builtin_isnan`）采用非算术方式进行检测。不过，该功能在不同目标架构和编译器版本上的支持程度存在差异，使用前建议查阅对应平台的实现文档。

## 七、Payload 的应用

NaN 的尾数部分除标志位外，剩余比特位（payload）可用于编码额外的信息。例如，苹果公司的 SANE（Standard Apple Numerics Environment）数学库曾利用 NaN payload 来编码具体的错误类型，使得一个“非数”同时承载了“为什么它不是数”的上下文信息。

该用法并未被 IEEE 754 标准强制规定，属于实现层面的可选扩展。不同操作系统和运行时库对 payload 的读写接口支持度不一，在跨平台代码中需谨慎使用。

## 总结

| 特性          | Quiet NaN（QNaN） | Signaling NaN（SNaN） |
| ----------- | --------------- | ------------------- |
| 尾数最高位（主流架构） | 1               | 0                   |
| 运算时行为       | 不触发异常，结果传播      | 触发无效操作异常            |
| 主要应用场景      | 表示无效运算的返回值      | 标记未初始化或诊断数据         |
| 实际使用频率      | 高               | 低                   |
