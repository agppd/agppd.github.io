## 一、许可证的基本作用

许可证是一份法律文本，依据版权法授予用户原本被保留的权利。若作品未附带许可证，则默认适用“所有权利保留”，他人不得复制、修改或分发。

## 二、开源协议

开源协议允许用户查看、使用、修改和分发软件的源代码。常见的开源协议包括以下类型。

### 1. GPL（GNU General Public License）
- 核心条款：衍生作品若对外分发，必须采用相同许可证（即“传染性”）。
- 典型应用：Linux 内核、Git。
- 适用场景：要求所有派生作品保持开源状态。

### 2. MIT License
- 核心条款：仅需保留版权声明，允许任意使用、修改、闭源分发及商业利用。
- 典型应用：React、Node.js、jQuery。
- 适用场景：希望代码被最广泛采用，不限制闭源商用。

### 3. Apache License 2.0
- 核心条款：在 MIT 基础上增加明确的专利授权及商标限制条款。
- 典型应用：Kubernetes、TensorFlow、Android。
- 适用场景：企业级项目，涉及专利风险管理和品牌保护。

### 4. BSD License
- 核心条款：与 MIT 类似，但对使用原作者名义进行推广有更严格限制。
- 典型应用：Go 语言、Nginx（早期版本）。

### 5. LGPL（GNU Lesser General Public License）
- 核心条款：允许闭源软件通过动态链接方式调用本库，但修改库本身后分发仍需开源。
- 典型应用：GTK、FFmpeg（部分组件）。

## 三、闭源协议

闭源协议即专有软件许可证，用户通常仅获得二进制形式的运行权，无权获取源代码、修改或重新分发。常见形式包括：

- **商业许可证**：按用户数、设备数或订阅期付费。例如：Windows、Adobe Photoshop、Oracle 数据库。
- **共享软件**：免费试用有限时间或功能，完整版需付费。
- **免费软件**：免费使用但源代码不公开，且不允许修改。例如：微信、WinRAR（个人版）。

闭源协议的主要目的是保护知识产权及支撑商业销售模式。

## 四、CC 协议（知识共享协议）

CC 协议并非为软件设计，而是用于文字、图片、音频、视频、课件、数据集等非软件作品。它提供了从“完全版权”到“公有领域”之间的多种授权选项。

### 常用 CC 协议类型（从开放程度高到低排列）

| 标识 | 简称 | 核心要求 | 允许商业使用 | 允许修改 |
|------|------|----------|--------------|----------|
| CC0 | 放弃版权 | 作者放弃所有版权相关权利，作品进入公有领域 | 是 | 是 |
| BY | 署名 | 仅需署名 | 是 | 是 |
| BY-SA | 署名-相同方式共享 | 署名 + 修改后的作品须使用相同许可证 | 是 | 是（传染性） |
| BY-NC | 署名-非商业性 | 署名 + 禁止商业用途 | 否 | 是 |
| BY-ND | 署名-禁止演绎 | 署名 + 禁止修改（只能完整复制） | 是 | 否 |
| BY-NC-SA | 署名-非商业-相同方式共享 | 署名 + 非商用 + 相同许可证 | 否 | 是（传染性） |
| BY-NC-ND | 署名-非商业-禁止演绎 | 署名 + 非商用 + 禁止修改 | 否 | 否 |

> 注：除 CC0 外，所有 CC 协议均要求署名。“相同方式共享”(SA) 条款具有类似 GPL 的“传染性”。

### CC 协议与开源/闭源协议的区别

- **适用对象不同**：开源/闭源协议用于软件源代码或可执行文件；CC 协议用于创意内容（文字、图像、音视频等）。
- **法律基础一致**：均基于版权法，通过许可证授予额外权限。
- **不宜混用**：软件项目不应使用 CC 协议（CC0 除外偶尔用于数据集），内容作品也不建议使用开源许可证。

### 典型使用案例

- 博客文章：CC BY（允许转载，需署名）。
- 维基百科：所有文字采用 CC BY-SA（要求衍生作品保持相同许可）。
- 摄影作品：CC BY-NC-ND（仅允许非商业分享，禁止修改）。
- 开放数据/课件：CC0 或 CC BY（最大化可用性）。

> 注意：带有 NC（非商业）条款的 CC 协议对“商业使用”的定义在不同司法管辖区可能存在歧义。若期望作品被商业实体采用，建议避免使用 NC 条款。

## 五、三类协议的多维度对比

| 维度 | 开源协议 | 闭源协议 | CC 协议 |
|------|----------|----------|----------|
| 适用对象 | 软件源代码 | 软件可执行文件 | 文字、图片、音视频等 |
| 源代码/原稿可访问性 | 是 | 否 | 是（若分发原稿） |
| 是否允许修改 | 依协议（允许/有条件的允许） | 否 | 依协议（允许/禁止/SA 传染） |
| 是否允许商业使用 | 通常允许（包括 GPL） | 需付费或依授权条款 | 依协议（NC 条款禁止） |
| 典型代表 | MIT、GPL、Apache | Windows、Photoshop | CC BY、CC BY-SA |
| 责任担保 | 通常不提供担保（AS-IS） | 供应商通常承担质量责任 | 作者可按协议追究侵权 |

## 六、常见误解澄清

- **开源协议等于免费使用**：不成立。开源指自由而非价格免费，开源软件可以收取费用。
- **CC 协议等于无限制使用**：不成立。CC 协议通常要求署名，部分条款禁止商业用途或修改。
- **可以用 CC 协议发布软件**：不推荐。CC 协议未涵盖软件分发的技术细节（如专利、动态链接），软件应使用 OSI 认可的许可证。
- **闭源协议比开源协议更安全**：不绝对。安全性取决于开发与维护流程。开源协议允许社区审查，闭源协议依赖厂商内部流程。
- **CC0 放弃所有权利**：正确。作者不再主张版权，但商标权、专利权不受影响。

## 七、许可证选择指引

**软件项目：**
- 希望代码被广泛采用且允许闭源衍生 → MIT / BSD / Apache 2.0
- 要求衍生作品必须开源 → GPL / AGPL
- 允许闭源软件动态调用但修改需开源 → LGPL
- 核心资产需防止复制且依赖销售许可 → 闭源协议

**非软件作品（内容创作）：**
- 允许自由转载，仅需署名 → CC BY
- 要求衍生作品也采用相同许可 → CC BY-SA
- 禁止商业用途 → 增加 NC 条款
- 禁止修改 → 增加 ND 条款
- 放弃所有版权，进入公有领域 → CC0

## 八、总结

开源协议、闭源协议与 CC 协议分别服务于软件代码、软件产品及创意内容三种不同的作品类型。每种许可证在权限授予、修改条件、商业使用限制及责任分配上均有明确的法律规定。选择合适的许可证，是对作品版权状态的清晰声明，也是对使用者权利义务的规范界定。
