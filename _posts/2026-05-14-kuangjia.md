## 一、一个统一的存储程序模型

### 1.1 程序与数据的平权

在冯·诺依曼架构中，**指令和数据共用同一个线性地址空间**。这意味着：

- 指令本身可以被视为一种特殊数据：编译、加载、重定位、自修改代码（历史上）都依赖这一特性。
- 计算机无需在物理上区分“代码区”与“数据区”，只依靠程序计数器的上下文来定义“当前正在执行的指令流”。

这一思想的直接产物，便是**存储程序计算机**：一个能自动、通用地执行存储器中任意程序的机器。

### 1.2 顺序执行与指令周期

既然指令逐条存放在存储器中，机器就需要一种机制自动完成：  
“取出一条指令 $\longrightarrow$ 解读它要做什么 $\longrightarrow$ 执行它 $\longrightarrow$ 将结果放回去 $\longrightarrow$ 再取下一条指令”。

这就自然引出了一个**闭环的循环流程**，即指令周期。其设计目标是：

- **通用性**：无论哪种高级语言，最终都被翻译为这个循环可处理的机器指令序列。
- **顺序控制流**：程序计数器 `PC` 默认顺序递增，只有显式跳转才会打破这一流，这与人类思维的线性执行高度吻合。
- **状态隔离**：每条指令在其周期内是“原子”的，前一条指令的结果写入体系结构状态（寄存器、内存、标志）后，下一条指令才能观察到。

### 1.3 时钟同步与硬件分工

将指令周期划分为**取指、译码、执行、写回**四个阶段，并非技术上的唯一解，却反映了数字系统设计的两个关键原则：

- **时序划分**：每个阶段可以在一个或几个时钟周期内完成，便于将复杂操作切分为流水步。
- **功能分工**：不同阶段对应不同的硬件资源（存储器、译码器、运算器、寄存器堆），使得这些资源可以流水并行使用，为后来所有高性能设计奠定基础。

所以，这四阶段本质上是对“指令处理”这一事务的**逻辑解剖**，而非物理上的强制。（你学的是一门工科，不是历史！）

## 二、大体实现，即：如何用硬件搭建这个框架

下面我们将每一阶段映射到具体的硬件结构和控制逻辑。实现时，CPU 内部通常划分为**数据通路**和**控制器**两部分。

### 2.1 全局骨架：程序计数器 PC 与指令寄存器 IR

整个周期的推动者是**程序计数器**。PC 中永远存放着下一条待取指令的内存地址。  
**指令寄存器 IR** 则保存当前正在处理的那条指令的二进制编码。

启动或复位时，PC 被置为某个固定地址（如系统引导地址），随后周期便自行维系。

### 2.2 取指阶段：从内存搬运指令

**设计目标**：将 PC 所指向的内存单元内容读到 IR，并准备好下一个 PC。

**大体硬件实现**：

1. PC 的输出接到**存储器地址总线**，同时控制器发出**存储器读**信号。
2. 存储器将对应数据送到数据总线，被锁存进 IR。
3. **PC 增量逻辑**：一个专用加法器将当前 PC 加上当前指令的长度（定长或译码后确定），得到**顺序下一条地址**。该值经过多路选择器写回 PC。

> 若遇到分支或跳转，则多路器选择来自执行阶段计算出的目标地址，而非顺序增量。这一切换是控制器根据指令类型决定。

### 2.3 译码

**设计目标**：解析 IR 中的操作码和寻址方式，生成对应的控制信号，并从寄存器堆读取操作数。

**大体硬件实现**：

- **指令译码器**：组合逻辑或微程序 ROM，以 IR 的高位操作码为输入，输出数十个控制信号线（如 RegDst、ALUSrc、MemtoReg 等）。
- **寄存器文件**：双读端口，由指令中寄存器号字段寻址，同时输出两个源操作数。
- **立即数扩展单元**：将指令中的短立即数字段符号或零扩展为完整宽度。
- **控制单元**汇总所有控制信号，形成一个**控制字**，去指挥执行阶段各部件。

对于复杂指令集，一条指令可能被译码为内部多个微操作，此时需要增加微序列器。但整体逻辑仍然是将“指令意图”翻译为微操作序列。

### 2.4 执行

**设计目标**：利用 ALU 完成算术逻辑运算，或为访存计算出有效地址。

**大体硬件实现**：

- **算术逻辑单元（ALU）**：接收两个输入，根据控制信号选择运算功能（加、减、与、或等），并输出结果和条件标志。
- **输入多路器**：选择 ALU 的第一个操作数来自寄存器还是 PC（用于分支计算）；第二个操作数来自寄存器还是立即数（由译码阶段控制）。
- **分支地址计算**：通常将 PC 值与偏移量相加，产生分支目标地址。

对于加载/存储指令，执行阶段的主要任务就是用 ALU 计算基址 + 偏移量，形成**有效地址**，该地址随即将被送到数据存储器。

### 2.5 写回

**设计目标**：将执行阶段产生的结果更新到架构寄存器或内存，完成指令可见效果。

**大体硬件实现**：

- **多路选择器**（MemtoReg）：选择写回寄存器的数据是 ALU 的结果，还是从数据存储器读出的值。
- **寄存器文件写端口**：在时钟沿，根据控制信号和目的寄存器号，将选中的数据写入。
- **存储器写信号**：对于存储指令，执行阶段会将数据写入数据存储器，写回阶段可能仅确认完成或处理写缓冲。

需注意，写回阶段**不允许修改还没有被下一条指令读取的寄存器**，这需要流水线互锁机制来保证，但基本的单周期实现中写回仅是简单的写使能。

## 三、框架的并行化实现

**BY DS**

尽管我们的逻辑划分是四个阶段，但实现上可以设计为**单周期、多周期或流水线**三种风格：

- **单周期实现**：整个指令在一个长时钟周期内完成，所有阶段组合成一条数据通路，简单但时钟频率受限于最慢指令。
- **多周期实现**：每个阶段占用一个时钟周期，可以复用 ALU 和存储器，但每周期只执行一条指令。
- **流水线实现**：将四阶段（或五阶段，访存独立）切分为流水线寄存器分隔的站，指令重叠执行。这完全是建立在我们划分的阶段基础之上的。

无论哪种方式，核心框架都没有变：**取指→译码→执行→写回**这一抽象序列始终存在，只不过在流水线中它们在时间上交错。

总体流程：

```plaintext
                            [指令周期框架]
                                 |
        +------------------------+------------------------+
        |                        |                        |
        v                        v                        v
   [取指阶段]               [译码阶段]             [执行+写回阶段]
                                                      (含访存)

+----+      +------+      +-----------+     +-----+     +--------+
| PC |----->| IMEM |      | 控制单元   |     | ALU |     | DMEM   |
+----+      +------+      +-----------+     +-----+     +--------+
  |             |               ^              ^  |          ^
  |             v               |              |  |          |
  |          +-----+            |              |  |          |
  |          | IR  |------------+              |  |          |
  |          +-----+                           |  |          |
  |             |                              |  |          |
  |             v (寄存器号/立即数)             |  |          |
  |        +----------+                        |  |          |
  |        | 寄存器堆  |---- 读数据 -----------+  |          |
  |        +----------+                           |          |
  |                     <---- 写回结果 ------------+          |
  |                                                          |
  +<------- 分支/跳转目标地址 (来自 ALU 或 PC+偏移) ---------+
```
