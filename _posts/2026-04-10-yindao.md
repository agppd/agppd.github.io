## 计算机引导流程概述（BIOS 与 UEFI）

计算机引导过程指从硬件加电到操作系统内核获得控制权的完整序列。根据固件标准不同，该过程可分为 **传统 BIOS 引导** 与 **UEFI 引导** 两类实现路径。

### 阶段一：上电与固件执行

**1. 电源启动与复位向量**

- 电源向主板供电，系统复位信号释放后，CPU 各寄存器被置为预定义初始状态。
- CPU 指令指针（IP / RIP）被硬连线指向特定物理地址，该地址称为**复位向量**。
  - 传统 x86 实模式下，复位向量地址为 `0xFFFF:0xFFF0`（物理地址 `0xFFFF0`）。
  - UEFI 固件环境下，复位后 CPU 通常仍从该传统地址开始执行，随后通过固件代码切换至保护模式或长模式。

**2. 固件初始化与硬件自检**

- 主板上的固件（BIOS 或 UEFI）存储于 SPI Flash 芯片中，复位后首先被执行。
- **传统 BIOS 执行流程：**
  - 执行**上电自检**（POST，Power-On Self Test），依次检查基础硬件设备（内存、键盘、显示适配器、磁盘控制器等）的可用性。
  - 硬件检测为串行执行，错误时通过蜂鸣声代码或端口 80h 诊断卡输出状态。
  - 自检完成后，BIOS 初始化中断向量表（IVT）和 BIOS 数据区（BDA），提供基本的 16 位实模式服务调用接口。
- **UEFI 固件执行流程：**
  - UEFI 遵循平台初始化（PI，Platform Initialization）规范，执行阶段包括：
    - **SEC（Security）**：处理复位向量与缓存初始化。
    - **PEI（Pre-EFI Initialization）**：初始化内存控制器与基础硬件，建立最小执行环境。
    - **DXE（Driver Execution Environment）**：加载并执行硬件驱动，支持并行初始化，缩短固件执行时间。
  - POST 功能在 UEFI 中分散于 PEI 与 DXE 阶段，错误信息可通过图形界面或串行控制台输出。

### 阶段二：引导设备选择与加载

**1. 引导设备枚举**

- 固件根据 NVRAM 存储的引导顺序变量（BootOrder）或默认规则，枚举可引导设备列表（硬盘、USB 存储设备、光驱、网络接口等）。
- BIOS 通过 INT 13h 磁盘服务访问块设备；UEFI 通过**块 I/O 协议**（Block I/O Protocol）以逻辑块地址（LBA）方式访问设备。

**2. 引导代码或引导文件的加载**

- **BIOS 引导路径：**
  - BIOS 读取候选引导设备的**第一个物理扇区**（主引导记录，MBR，长度 512 字节）。
  - 检查扇区末尾两字节是否为签名值 `0x55 0xAA`。若匹配，BIOS 将该扇区内容复制到物理地址 `0x7C00`，并将控制权转移至该地址开始执行。
  - MBR 代码空间仅 446 字节（扣除分区表与签名），通常无法直接加载操作系统内核。MBR 代码的主要任务是：
    1. 解析自身分区表，定位标记为活动（Active）的主分区。
    2. 加载该分区的**卷引导记录**（VBR，Volume Boot Record，亦称分区引导扇区）。
  - VBR 包含文件系统识别代码，可读取该分区内操作系统引导加载器（如 Windows 的 `bootmgr`、Linux 的 `GRUB stage2` 文件）。
  - 引导加载器随后切换 CPU 至保护模式或长模式，加载内核映像并移交控制权。

- **UEFI 引导路径：**
  - UEFI 固件内置对 **FAT 文件系统**（FAT12/16/32）的驱动支持，不依赖固定扇区地址。
  - UEFI 通过引导管理器读取**EFI 系统分区**（ESP，EFI System Partition）内的可执行文件。
    - ESP 分区 GUID 为 `C12A7328-F81F-11D2-BA4B-00A0C93EC93B`，通常格式化为 FAT32。
  - 根据引导变量（如 `Boot###` 条目）指定的路径，UEFI 直接加载并执行 `.efi` 格式的可执行文件。
  - 缺省引导文件路径为 `\EFI\BOOT\BOOT{ARCH}.EFI`，其中 `{ARCH}` 为架构标识（如 x64 对应 `BOOTX64.EFI`）。
  - `.efi` 程序以 **PE/COFF+** 格式存在，UEFI 固件将其加载至内存后，以 UEFI 应用程序身份执行。此时 CPU 已处于保护模式或长模式，`.efi` 程序可通过 UEFI 引导服务（Boot Services）访问文件系统、网络、控制台等资源。

### 阶段三：操作系统加载与内核初始化

**1. 引导管理器与内核加载**

- 以 Windows 引导为例：
  - UEFI 环境下，UEFI 固件执行 `\EFI\Microsoft\Boot\bootmgfw.efi`（Windows 引导管理器）。
  - BIOS 环境下，VBR 加载并执行 `bootmgr` 文件（无扩展名），该文件位于活动分区根目录。
- 引导管理器读取**引导配置数据**（BCD，Boot Configuration Data），确定要启动的操作系统条目及对应内核文件路径。
- 加载器（Windows 中为 `winload.efi` 或 `winload.exe`）负责：
  - 读取并校验内核文件 `ntoskrnl.exe` 与 HAL 层模块。
  - 加载系统注册表配置单元与核心设备驱动。
  - 准备内存描述结构并调用固件退出接口。

**2. 退出固件服务与内核接管**

- BIOS 引导路径中，引导加载器在执行内核代码前调用 BIOS 中断切换显示模式、获取内存映射，随后禁用中断并将 CPU 切换至保护模式。内核启动后，BIOS 中断服务不再被使用。
- UEFI 引导路径中，操作系统加载器通过调用 `ExitBootServices()` 函数通知固件终止引导服务。此后控制权完全移交操作系统内核，UEFI 运行时服务（Runtime Services）仍可被操作系统调用（例如用于访问 NVRAM 变量或更新固件）。

**3. 操作系统初始化**

- 内核初始化阶段包括：初始化进程管理与调度器、内存管理器、I/O 管理器、加载剩余驱动程序、启动系统服务。
- 初始化完成后，启动用户会话管理进程（Windows 为 `smss.exe` 与 `winlogon.exe`），最终显示登录界面。

## BIOS 与 UEFI 引导机制对比表

| 比较项 | BIOS 引导 | UEFI 引导 |
| :--- | :--- | :--- |
| **引导代码定位方式** | 读取固定物理扇区（LBA 0 的 MBR） | 访问 ESP 分区内文件系统路径下的 `.efi` 文件 |
| **分区表格式支持** | MBR（主引导记录）分区表 | GPT（GUID 分区表）为主，兼容 MBR |
| **固件运行时模式** | 16 位实模式，引导加载器切换至保护模式 | 32 位或 64 位保护模式/长模式，无需模式切换 |
| **文件系统感知** | 固件不识别文件系统，依赖 VBR 代码解析 | 固件内置 FAT 驱动，可直接读取 ESP 内文件 |
| **安全启动** | 不支持 | 支持 Secure Boot，基于数字签名校验 `.efi` 文件 |
| **启动性能** | 串行硬件初始化，速度较慢 | 并行硬件初始化，支持 Fast Boot 缩短固件执行时间 |
| **平台兼容性** | 仅支持传统 PC/AT 架构 | 适用于 IA-32、x64、ARM、RISC-V 等多架构平台 |
