## 一、未定义行为

### 1.1 什么是未定义行为？

未定义行为发生在你违反C++语言某些规则的时候。这意味着**编译器被允许对你的代码做任何事情**——它可能崩溃、可能静默地产生错误结果、也可能“恰好”在你机器上正常工作。

UB最危险的地方不在于“会崩”，而在于**它往往什么都不做**：代码在你机器上跑得好好的，上线后却在某个版本、某个编译选项、某台机器上突然炸掉。

C++标准将程序行为分为几类：
- **良构（well-formed）** ：行为被精确定义
- **实现定义行为**：因实现而异，但实现必须文档化
- **未指定行为**：因实现而异，无需文档化
- **未定义行为**：没有任何限制，程序可能做任何事情
- **错误行为（erroneous behavior，C++26起）** ：实现被推荐诊断的非正确行为

——By DS

### 1.2 常见的UB（及处理办法）

**（1）越界访问**

```cpp
int arr[5] = {0};
for (int i = 0; i <= 5; ++i) {
    arr[i] = i;  // i=5时越界，UB
}
```

越界访问可能覆盖相邻变量、可能什么都不发生、也可能直接崩溃。

**处理办法**：优先使用`std::array`/`std::vector`替代裸数组；开启AddressSanitizer（ASan）检测；编译时加`-fsanitize=address`。

**（2）悬垂引用/悬垂指针**

```cpp
int& foo() {
    int x = 42;
    return x;  // 返回局部变量的引用——悬垂引用！
}

std::string_view getName() {
    std::string s = "Alice";
    return s;  // string_view不拥有数据，返回即悬空
}
```

引用、指针、`string_view`都不管理生命周期。

**处理办法**：不要返回指向局部对象的引用或视图；接口层明确“谁拥有数据”；必要时宁可拷贝。

**（3）未初始化变量**

```cpp
int x;
if (x > 0) {  // 读取未初始化的x，UB
    doSomething();
}
```

在优化级别提高后，编译器可能直接假设这种情况**永远不会发生**，然后大胆删代码。

**处理办法**：声明即初始化；打开`-Wall -Wextra -Wuninitialized`。

**（4）严格别名违规**

```cpp
float f = 1.0f;
int* p = reinterpret_cast<int*>(&f);
int x = *p;  // UB：通过不相关类型指针访问对象
```

除了`char`/`unsigned char`/`std::byte`，通过不相关类型指针访问对象都违反严格别名规则。

**处理办法**：使用`std::memcpy`做类型重解释；C++20之后优先用`std::bit_cast`。

**（5）有符号整数溢出**

```cpp
int x = INT_MAX;
x += 1;  // UB！
```

**无符号**整数溢出是定义良好的（模2ⁿ回绕），但**有符号**整数溢出是UB。

**处理办法**：需要回绕行为时使用无符号类型；需要检测溢出时使用`__builtin_add_overflow`等编译器内置函数。

### 1.3 运行时错误与调试策略

运行时错误包括内存泄漏、空指针解引用、数组越界等。除上述UB外，常见运行时错误还有：

| 错误类型 | 典型表现 | 调试手段 |
|---------|---------|---------|
| 段错误（Segfault） | 访问非法内存 | GDB + core dump |
| 内存泄漏 | 内存持续增长 | Valgrind、ASan |
| 栈溢出 | 无限递归 | GDB查看调用栈 |
| 死锁 | 程序卡死 | GDB多线程调试 |

## 二、GDB底层调试

### 写在前面
当`-g -O0`已经就绪，GDB就不再是“断点查看器”，而是**操作系统给我们的进程查看器**。

> 注：这部分有的东西作者本人都没用过，大量依赖网上的资料，请酌情使用

### 2.1 寄存器级单步：`stepi` 与 `nexti`
高级语言的单步（`s`/`n`）步进太大，难以发现细枝末节的问题。底层调试必须用`si`（汇编级步入）和`ni`（汇编级步过），每次执行一条CPU指令。

```gdb
(gdb) display/i $pc          # 每次停住时自动显示当前指令（$pc是程序计数器）
(gdb) si                     # 执行当前$pc指向的一条机器指令
(gdb) info registers         # 查看所有通用寄存器（rax, rbx, rsp, rbp, rip...）
(gdb) p/x $rax               # 单独查看RAX寄存器的16进制值
```

### 2.2 内存布局探针
`x`直接读取**虚拟内存地址**，可以快速的查看内存上的问题。

```gdb
(gdb) x/20xg $rsp            # 以8字节(64位)为单位，打印栈顶rsp往上的20个内存单元
(gdb) x/10i 0x400500         # 将0x400500地址开始的10个字节解析为汇编指令
(gdb) x/s $rdi               # 将RDI寄存器指向的位置当作字符串打印（常用于Syscall参数）
(gdb) x/wx &global_var       # 查看全局变量的4字节原始值
```

### 2.3 使用：`info frame` 与 `frame` 查看栈帧底层信息
不仅要看调用栈（`bt`），更要看**栈帧的物理布局**：

```gdb
(gdb) info frame             # 显示当前栈帧的底层细节
```
输出包含：
- `stack level`：栈帧层级
- `pc`：该帧的返回地址保存在哪里
- `saved registers`：哪些寄存器被保存在栈上（如RBP、RBX）
- `frame at`：该帧的基址（`$rbp`的值）
- `Arglist at`：参数在栈上的位置（针对System V ABI）

**进阶场景**：`frame <n>`切到指定帧后，用`info frame`对比`$rsp`和`$rbp`的差值，手工推算局部变量在栈上的偏移量，甚至可以通过`set {int}($rbp-8)=100`直接**修改栈上的局部变量**，这种可以块速进行调试，极大的避免了编译型语言重复编译，重新开始调试的问题。

### 2.4 虚函数表透视
C++多态在底层就是通过**对象首地址的虚表指针**实现的。用GDB直接解剖：

```cpp
class Base { virtual void foo(){} };
Base* p = new Derived();
```
```gdb
(gdb) p *p                    # 看对象内容，第一项就是_vptr
(gdb) x/gx p                  # 读取p指向的前8字节，得到虚表地址
(gdb) x/gx 0x(vtable_addr)    # 查看虚表内容，前两个一般是偏移量，后面是指向函数的指针
(gdb) x/i 0x(func_addr)       # 将虚表中的函数地址反汇编为指令
```

### 2.5 捕获底层信号与异常抛出点
程序崩溃本质是CPU触发异常（如#PF缺页）被内核转为`SIGSEGV`信号。GDB可捕获信号的**最初发源地**：

```gdb
(gdb) catch signal SIGSEGV    # 在段错误信号发出的那一刻挂起（早于core dump）
(gdb) catch throw             # 捕获C++ throw抛出的异常（实际调用__cxa_throw）
```
配合`bt`和`info registers`，能在栈被破坏前定格现场，查看`$rip`（指令指针）指向的具体非法汇编指令。

### 2.6 应对优化（`-O2`）下的调试：
高优化下变量被优化进寄存器或直接消除。此时采用**寄存器盯梢**：

```gdb
(gdb) info args               # 即使显示optimized out，尝试info registers
(gdb) disas                   # 反汇编当前函数，人工推理该变量被映射到哪个寄存器
(gdb) p $rax                  # 强行打印寄存器当前值
```
另外，GDB 7.0+支持`-g -Og`（优化调试平衡），生成更“可跟踪”的优化代码。

## 三、C++设计哲学：零成本抽象与RAII

**不要觉得这些是废话，理解这些才能最大程度的明白编译器会给你补充什么以及它怎么理解你的代码**

### 3.1 C++的核心设计哲学

C++的设计追求**性能**和**可移植性**。其核心理念包括：

**零成本抽象（Zero-Cost Abstraction）** ：你不需要为没有使用的特性付出代价，使用高级抽象不应比手写底层代码更慢。

**多范式编程**：C++同时支持面向对象、泛型、函数式和过程式编程。开发者应根据问题域灵活选择最合适的范式。

**不为你不需要的东西付出代价**：C++不会强制你使用垃圾回收、运行时类型信息等可能带来性能开销的特性。

此时，你应该明白了`Double Free`~~（双倍的自由）~~产生的原因：你没有使用深拷贝，编译器理解了，故而默认使用浅拷贝，所以处理不当就会产生，所以解决方式之一是使用深拷贝。

### 3.2 RAII：C++的生存法则

——By DS

RAII（Resource Acquisition Is Initialization，资源获取即初始化）是C++的**基石**：

> **资源的生命周期应绑定在对象的生命周期内。** 

构造函数获取资源，析构函数释放资源——编译器保证无论函数如何退出（正常返回、异常、goto、break），析构函数都会被调用。

```cpp
class FileHandle {
    FILE* f;
public:
    FileHandle(const char* path) : f(fopen(path, "r")) {}
    ~FileHandle() { if (f) fclose(f); }  // 自动释放
    // 禁止拷贝，支持移动...
};
```

RAII不仅是内存管理（智能指针），更是**一切资源**（文件句柄、锁、网络连接、数据库连接）的管理哲学。

### 3.3 “易于正确使用，难以错误使用”

C++设计强调API的可用性：好的接口应该让正确用法自然、错误用法困难。这也是为什么现代C++推荐使用`std::unique_ptr`而非裸指针——它显式表达了所有权语义，让资源管理变得安全且零开销。

## 四、C++编译原理（大体）

一个`.cpp`文件变成可执行程序需要经历**四个阶段**：

### 4.1 预处理（Preprocessing）

处理以`#`开头的指令：
- 展开宏（`#define`）
- 包含头文件（`#include`）
- 删除注释
- 条件编译（`#ifdef`、`#ifndef`）

```bash
g++ -E main.cpp -o main.i    # 只做预处理
```

### 4.2 编译（Compilation）

将预处理后的代码转换为**汇编语言**：
- 词法分析、语法分析、语义分析
- 生成中间代码
- 优化
- 生成汇编代码

```bash
g++ -S main.cpp -o main.s    # 生成汇编代码
```

### 4.3 汇编（Assembly）

将汇编代码转换为**机器码**（目标文件`.o`或`.obj`）：

```bash
g++ -c main.cpp -o main.o    # 生成目标文件
```

这里说一下：目前大部分用的都是`.o`文件，作者上次看见`.obj`文件还是在`Intel 16位汇编`里使用`MS-DOS`上的`NASM`编译生成的文件里看到的`.OBJ`文件。

### 4.4 链接（Linking）

将所有目标文件和库合并为最终可执行文件：
- 符号解析
- 地址绑定
- 合并段（section）

```bash
g++ main.o util.o -o myprogram
```

头文件保护符（`#ifndef`）的根源就在链接阶段——为避免符号重复定义而产生。

如果你在不修改系统栈与堆分配策略的情况下，申请了特别大的内存，则其实际报错是在这部分，因为`.bss`会直接炸掉

### 4.5 查看汇编

通过查看编译器生成的汇编代码，可以有效的解决部分在高级语言层面解决不了的问题，一些奇奇怪怪的报错大部分在汇编层面就会暴露

```bash
g++ -S main.cpp
```

然后查看`main.s`，那里就是汇编

## 五、C++部分语句本质


### 5.1 变量声明与初始化（栈与立即数）

```cpp
int a = 10;
```
- **底层**：在栈帧中预留4字节（`$rsp -= 4`，或使用红区`$rsp`不变），然后`mov DWORD PTR [rbp-4], 10`。`a`本质上是一个**编译器维护的栈偏移量别名**（`[rbp-4]`）。

### 5.2 算术与逻辑运算（寄存器暂存）
```cpp
int c = a + b;
```
- **底层**：`mov eax, DWORD PTR [rbp-4]`（加载a）; `add eax, DWORD PTR [rbp-8]`（加b）; `mov DWORD PTR [rbp-12], eax`（写回c）。**中间值永远活在寄存器中**，没有“变量”存放中间结果。

### 5.3 条件分支（`if` / `switch`）
```cpp
if (x > 0) { doA(); } else { doB(); }
```
- **底层**：`cmp DWORD PTR [rbp-4], 0`; `jle .L_else`（条件跳转）。`switch`在分支较多时可能优化为**跳转表（Jump Table）** ，本质是`jmp [QWORD PTR rax*8 + table_addr]` —— 地址即整数索引。

### 5.4 循环（`for` / `while`）
```cpp
while (i < n) { i++; }
```
- **底层**：循环体末尾插入`cmp`和`jl`（小于则回跳），构成**向后的条件跳转**。`for(;;)`直接退化为`jmp`无条件跳转。

### 5.5 函数调用（调用约定与栈帧）

> 这部分是DS写的，实在是有点难，作者较不准这部分的准确性。

```cpp
int ret = foo(a, b);
```

在 **32位 x86（IA-32）** 架构下，Linux/GCC 默认使用 **cdecl** 调用约定（也有 `-mregparm` 等变体，但此处以标准 cdecl 为准）：

#### （1）参数传递：全栈式压入
- **所有参数**（无论整数、指针）均通过 **栈（stack）** 传递，不使用寄存器传参。
- 压栈顺序为 **从右到左**：先压入最后一个参数（`b`），再压入倒数第二个（`a`），依此类推。
- 调用前，调用者（caller）负责将参数压栈，并调整 `%esp`（栈指针）。

汇编伪代码（Intel 语法）：
```nasm
push b          ; 先压入最后一个参数
push a          ; 再压入第一个参数
call foo        ; 压入返回地址，跳转到 foo
add esp, 8      ; 调用者清理栈上的两个参数（cdecl 约定）
mov ret, eax    ; 返回值在 EAX 中
```

#### （2）`call` 指令与返回地址
- `call foo` 执行时，CPU 将 **当前指令指针（`%eip`）的下一条地址**（返回地址）压入栈顶（`%esp -= 4; *%esp = return_addr`），然后跳转到 `foo` 的入口地址。
- 因此，进入被调用函数（callee）时，栈顶（`%esp`）指向返回地址，紧接着下方（`%esp+4`）是最后一个参数（`b`），再往下（`%esp+8`）是第一个参数（`a`）。

#### （3）栈帧构建
被调用函数 `foo` 内部，通常先建立标准栈帧：
```nasm
foo:
    push ebp          ; 保存调用者的 EBP
    mov ebp, esp      ; 设置当前栈帧基址
    sub esp, N        ; 为局部变量分配 N 字节空间（N 由编译器静态计算）
    ; ... 函数体 ...
    mov esp, ebp      ; 恢复 ESP（或直接 leave）
    pop ebp
    ret               ; 弹出返回地址并跳转
```
- `%ebp` 现在固定指向当前栈帧底部（高地址），局部变量可通过 `[ebp-4]` 等负偏移访问。
- **参数访问**：`[ebp+8]` 为第一个参数 `a`，`[ebp+12]` 为第二个参数 `b`（因为返回地址占 4 字节，原 `ebp` 占 4 字节）。
- 局部变量分配在 `[ebp-N]` 到 `[ebp-4]` 之间。

#### （4）返回值传递
- 整数 / 指针返回值通过 **`%eax` 寄存器**（32位）带回。
- 调用者从 `%eax` 读取返回值并赋给变量 `ret`。

#### （5）调用者清理 vs 被调用者清理
- **cdecl** 约定要求 **调用者** 在 `call` 返回后清理栈上的参数（`add esp, 参数总字节数`）。
- 另一种常见约定 **stdcall**（Windows API）则由被调用者通过 `ret n` 清理，但 Linux 下极少使用，此处不展开。

#### （6）快速寄存器传参（非标准）
- GCC 支持 `-mregparm=3` 选项，允许前三个整数参数通过 `EAX, EDX, ECX` 传递，但这属于拓展，非标准 ABI。我们的讨论基于默认 cdecl。

---

> **本质**：函数在 32 位 x86 下退化为 **栈上的数据搬运与跳转指令**。函数名在汇编层面被解析为 **立即地址**（或 `call` 的重定位符号）。`call/ret` 指令携手完成控制流转移，而 `%ebp` 作为“锚点”让编译器能够用固定偏移访问参数和局部变量。没有“栈帧”的硬件概念，它只是约定俗成的寄存器用法——这一切都显露在 `objdump -M intel -d` 的汇编输出中。

### 5.6 数组访问与下标运算
```cpp
arr[i] = 5;
```
- **本质**：`movsxd rax, DWORD PTR [rbp-4]`（取i）; `mov DWORD PTR [rbp + rax*4 - 16], 5`。`arr`被编译器替换为栈基址`rbp`或数据段基址，下标`i`转译为**缩放因子** 的寻址模式。这就是“数组名是指针常量”的真正落地。

> 缩放因子是下标值与元素字节大小的乘积，其目的是将逻辑上的“第 i 个元素”转换为内存中的**字节偏移量**。

### 5.7 结构体/类成员访问（偏移量计算）
```cpp
obj.member = 3;
```
- **底层**：`obj`地址固定，`member`在编译期被赋予固定偏移量（如`+8`）。汇编为`mov DWORD PTR [rdi+8], 3`（`this`传入`RDI`）。成员变量本质就是**基址+编译期常量偏移**。

### 5.8 虚函数调用（间接跳转）
```cpp
p->foo();
```
- 若`foo`为虚函数，汇编为：`mov rax, [rdi]`（取虚表指针）; `mov rax, [rax + 8]`（取虚表中第二个函数指针，偏移取决于`foo`在vtable中的索引）; `call rax`。这是**通过寄存器进行的间接跳转**，与普通`call 0x400500`（直接跳转）截然不同。

### 5.9 引用（`int& ref = a;`）
- **本质**：引用在底层**完全等同于指针常量**。它本身不占用空间（除非编译器必须分配），使用时`ref`会被直接替换为解引用`a`的地址。汇编代码与`int* const p = &a; *p`一模一样。

### 5.10 Lambda 表达式
```cpp
auto f = [x](int y){ return x + y; };
```
- 无捕获的lambda退化为普通函数指针。
- 有捕获的lambda编译为**仿函数（Functor）** 的匿名类实例。`operator()`的调用本质上就是调用该类内部的成员方法，`this`指向捕获列表所在栈对象。捕获变量成为该匿名类的成员变量，在构造时初始化。

### 5.11 `auto` 类型推导
- **运行时零开销，无底层痕迹**。`auto`只发生在编译期类型推导，生成的汇编代码与你手工写的类型完全一致，属于**纯粹的编译期语法糖**。

### 5.12 `new` / `delete` 与 `malloc` / `free`
- `new`底层先调用`operator new`（通常包装`malloc`）分配堆内存，然后**插入构造函数调用**。
- `delete`先调用析构函数，再调用`operator delete`（`free`）。
- 堆地址存放在`RAX`，类对象的内存布局中，虚表指针（如果有）由构造函数在堆内存开头写入。

### 5.13 异常处理（`try` / `throw` / `catch`）
- `throw`底层调用`__cxa_throw`（Itanium ABI），它会展开栈（Unwind），查找匹配的`catch`处理程序。这一过程依赖**异常处理表（LSDA，Language-Specific Data Area）** ，位于只读段。`try/catch`并不产生常规的跳转指令，而是注册一个“着陆垫（Landing Pad）”，异常抛出时通过`_Unwind_`库函数回退栈帧。

## 总结：从语言到机器的映射

——By DS

| C++语句 | CPU/内存本质 |
| :--- | :--- |
| 变量 | 栈偏移量或寄存器别名 |
| 运算符 | 寄存器算术、位移、逻辑指令 |
| 条件/循环 | `cmp` + 条件跳转/无条件回跳 |
| 函数 | `call`/`ret` + 寄存器传参与返回约定 |
| 数组 | 基址 + 比例因子寻址（`base + index * scale`） |
| 结构体/类 | 基址 + 编译期固定偏移 |
| 虚函数 | 虚表指针索引 + 寄存器间接跳转（`call rax`） |
| 引用 | 解引用的指针常量（语法糖） |
| Lambda | 匿名仿函数类 + `operator()` |
| `auto` | 编译期类型擦除，运行时无痕迹 |
| `new/delete` | `malloc/free` + 构造/析构注入 |
| 异常 | 栈展开 + 查表跳转到着陆垫 |
