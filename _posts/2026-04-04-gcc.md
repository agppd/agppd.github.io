## 第 1 步：安装基础编译工具

**命令**（以 Ubuntu 为例）：
```bash
sudo apt install -y build-essential libmpc-dev libmpfr-dev libgmp-dev flex bison texinfo
```

**解释**：
- `build-essential`：这是 Ubuntu 中的一个元包，它会自动安装 `gcc`、`g++`、`make` 等最基本的编译工具。由于初始系统没有任何编译器，我们需要先用系统自带的软件包管理器安装一套可用的“种子编译器”。这套编译器虽然版本可能较旧，但足以用来编译新版本的 GCC。
- `libmpc-dev`、`libmpfr-dev`、`libgmp-dev`：分别是 MPC、MPFR、GMP 三个数学库的开发文件。GCC 在进行高精度算术运算（如编译时优化、浮点常量折叠）时需要它们。虽然 GCC 源码包中带有这些库的自动下载脚本，但提前通过包管理器安装可以避免后续手动编译这些库，节省时间。
- `flex`、`bison`：词法分析器和语法分析器生成工具。GCC 的语法解析部分由它们生成代码。
- `texinfo`：生成 GCC 文档（info 格式）所需的工具。如果不需要文档，也可以跳过，但安装它可避免配置阶段警告。

**注意**：对于 CentOS/RHEL，`Development Tools` 组等价于 `build-essential`，而 `gmp-devel` 等则是对应的开发包。

## 第 2 步：下载 GCC 源码并准备依赖

**命令**：
```bash
wget https://ftp.gnu.org/gnu/gcc/gcc-14.2.0/gcc-14.2.0.tar.xz
tar -xf gcc-14.2.0.tar.xz
cd gcc-14.2.0
./contrib/download_prerequisites
cd ..
```

**解释**：
- `wget`：从 GNU 官方镜像下载 GCC 源代码压缩包。选择版本 14.2.0 作为示例，你可以替换为其他版本号。
- `tar -xf`：解压源码包（`xz` 格式需要 `tar` 自动识别）。
- `./contrib/download_prerequisites`：这是 GCC 源码自带的便利脚本。它会自动下载 GMP、MPFR、MPC 等依赖库的源码，并解压到 GCC 源码目录下的对应子目录中。这样做的好处是：在配置 GCC 时，构建系统会自动链接这些本地副本，无需手动安装到系统目录，也避免了系统库版本不匹配的问题。该脚本还会验证下载文件的完整性。
- 最后 `cd ..` 回到上级目录，为下一步的“外部构建”做准备。

## 第 3 步：配置构建

**命令**：
```bash
mkdir build-gcc-14.2.0 && cd build-gcc-14.2.0
../gcc-14.2.0/configure \
    --prefix=/opt/gcc-14.2.0 \
    --enable-languages=c,c++ \
    --disable-multilib
```

**解释**：
- **外部构建（out‑of‑source build）**：新建一个 `build-gcc-14.2.0` 目录并在其中运行 `configure`。这是 GCC 官方强烈推荐的做法。它将所有编译生成的临时文件（`.o` 文件、中间产物等）与源代码目录分开，便于清理、维护以及同时构建多个配置版本。
- `../gcc-14.2.0/configure`：执行源码目录中的配置脚本，相对路径 `..` 指向上一级。
- `--prefix=/opt/gcc-14.2.0`：指定安装目录。这里选择 `/opt/gcc-14.2.0`，而不是默认的 `/usr/local`。主要原因是便于多版本共存：每个版本的 GCC 安装在独立的目录下，之后可以通过 `update-alternatives` 或环境变量轻松切换。
- `--enable-languages=c,c++`：只构建 C 和 C++ 语言前端。你可以根据需要添加 `fortran`、`objc`、`go` 等。保留不需要的语言前端可以减少编译时间和最终安装体积。
- `--disable-multilib`：禁用多库支持。默认情况下 GCC 会同时生成 32 位和 64 位版本的库（如果目标平台支持）。禁用后只生成与宿主机相同的位数（通常是 64 位），可以显著缩短构建时间并减少磁盘占用。如果你的应用需要编译 32 位程序，则不应使用此选项。

**其他常用配置选项**：
- `--enable-checking=release`：在发布版级别进行内部一致性检查。默认就是 `release`，通常无需显式指定。若调试需要，可改为 `--enable-checking=yes` 或 `--enable-checking=assert`。
- `--disable-bootstrap`：禁用三阶段自举。默认启用自举，它会用旧编译器编译新编译器，再用新编译器重新编译自身，确保编译结果可靠。禁用可节省时间，但只建议在测试或交叉编译时使用。

## 第 4 步：编译与安装

**命令**：
```bash
make -j$(nproc)
sudo make install
```

**解释**：
- `make -j$(nproc)`：启动编译。`$(nproc)` 返回 CPU 核心数，`-j` 参数启用并行编译。例如四核 CPU 上 `-j4` 或 `-j8`（超线程）。并行编译可以大幅缩短总耗时（从 2‑3 小时降到 30‑60 分钟）。注意：如果系统内存较小，并行度过高可能导致内存不足，可适当降低 `-j` 值。
- **自举（bootstrap）**：执行 `make` 时，默认会进行三阶段自举。具体过程：
  - **Stage 1**：用系统自带的旧 GCC 编译一个新 GCC（功能受限）。
  - **Stage 2**：用 Stage 1 的 GCC 再次编译 GCC 源码。
  - **Stage 3**：用 Stage 2 的 GCC 再次编译 GCC 源码，并与 Stage 2 的结果进行比较。如果两者一致，说明编译器可信。这个过程保证了最终编译器的正确性和健壮性。
- `sudo make install`：将编译好的二进制文件、库文件、头文件等复制到 `--prefix` 指定的 `/opt/gcc-14.2.0` 目录下。需要 `sudo` 是因为 `/opt` 通常需要 root 权限写入。

**验证安装**：
```bash
/opt/gcc-14.2.0/bin/gcc --version
```
应显示新安装的 GCC 版本信息。

## 第 5 步：多版本管理

### 方法一：update-alternatives（Debian/Ubuntu）

**注册命令**：
```bash
sudo update-alternatives --install /usr/bin/gcc gcc /opt/gcc-14.2.0/bin/gcc 140 \
    --slave /usr/bin/g++ g++ /opt/gcc-14.2.0/bin/g++
```

**解释**：
- `update-alternatives` 是 Debian 系系统管理符号链接组的标准工具。它维护一个“替代项系统”，可以将一个通用命令名（如 `/usr/bin/gcc`）链接到多个候选版本之一。
- `--install` 参数：添加一个候选版本。
  - `/usr/bin/gcc`：主链接的路径（即系统命令的常用位置）。
  - `gcc`：替代项组的名称（同一组内的候选版本可以相互切换）。
  - `/opt/gcc-14.2.0/bin/gcc`：该候选版本的实际二进制文件路径。
  - `140`：优先级数字。当处于“自动模式”时，系统会选择优先级最高的版本。
- `--slave`：定义从属链接。它确保当切换 `gcc` 时，`g++` 也同步切换到同一目录下的对应版本，避免编译器版本不一致导致的 ABI 问题。你可以继续添加 `--slave` 来关联 `gcov`、`gfortran` 等。

**切换命令**：
```bash
sudo update-alternatives --config gcc
```
执行后会列出所有注册的 GCC 版本，输入数字即可切换。切换后，`/usr/bin/gcc` 和 `/usr/bin/g++` 的符号链接会指向新选择的版本。

### 方法二：alternatives（CentOS/RHEL）

原理与 `update-alternatives` 完全相同，只是命令名称不同（`alternatives`），用法基本一致。在 RHEL/CentOS 系统中，该工具由 `chkconfig` 包提供，通常已预装。

### 方法三：环境变量（临时切换）

**命令**：
```bash
export CC=/opt/gcc-14.2.0/bin/gcc
export CXX=/opt/gcc-14.2.0/bin/g++
```

**解释**：
- 许多构建系统（如 Autotools、CMake、Makefile）会检查环境变量 `CC`（C 编译器）和 `CXX`（C++ 编译器）。通过设置这些变量，可以临时覆盖系统默认的编译器。
- 这种方式**不修改系统全局设置**，只影响当前 Shell 会话及其启动的进程。适合单个项目、脚本或持续集成（CI）环境。
- 若希望永久生效，可将 `export` 命令写入 `~/.bashrc`、`~/.bash_profile` 或项目目录下的 `.env` 文件。

**使用示例**：
```bash
# 在当前 Shell 中编译项目
make CC=/opt/gcc-14.2.0/bin/gcc CXX=/opt/gcc-14.2.0/bin/g++

# CMake 项目
cmake -DCMAKE_C_COMPILER=/opt/gcc-14.2.0/bin/gcc \
      -DCMAKE_CXX_COMPILER=/opt/gcc-14.2.0/bin/g++ ..
```
